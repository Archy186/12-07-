**Задание 1. Резервное копирование**

Кейс

Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.

Необходимо описать, какие варианты резервного копирования подходят в случаях:

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

Приведите ответ в свободной форме.

**Решение:**

*1.1. Восстановление данных в полном объёме за предыдущий день
Для этого случая идеально подходит стратегия полных резервных копий (Full Backup). Это базовый и надежный подход.
Суть метода: Создается полная копия всей базы данных. Восстановление происходит в один этап из этого полного бэкапа.
Как часто делать: Ежедневно, в часы наименьшей нагрузки. Например, каждый день в 2:00 ночи.
Пример инструментов: Для PostgreSQL это pg_dump (логическая копия) или pg_basebackup (физическая копия). Для MySQL — mysqldump или mysqlbackup.
Плюсы: Простота восстановления. Минимальный риск ошибок.*
Минусы: Требует много места для хранения (каждый бэкап — это полный объем данных). Процесс может быть долгим для больших баз данных.

*1.2. Восстановление данных за час до предполагаемой поломки
Здесь одного полного бэкапа недостаточно. Нужна стратегия "Полный бэкап + Инкрементные бэкапы" или "Полный бэкап + Бинарные логи (WAL/Binlog)".
Суть метода:
Регулярно создаются полные бэкапы (например, раз в неделю).
Между ними создаются инкрементные бэкапы, которые содержат только изменения с момента последнего бэкапа (или предыдущего инкрементального). Для MySQL это может быть команда mysqlbackup --incremental, для PostgreSQL, начиная с версии 17, — pg_basebackup --incremental.
Альтернативно, вместо инкрементных бэкапов можно непрерывно архивировать журналы транзакций (WAL в PostgreSQL, Binary Log в MySQL). Это позволяет "накатить" все изменения, произошедшие после последнего полного бэкапа.
Процесс восстановления:
Восстанавливается последний полный бэкап.
Поверх него последовательно применяются все инкрементные бэкапы (или журналы транзакций) до нужного момента времени (за час до поломки).*

**Задание 2. PostgreSQL**

2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).
Приведите ответ в свободной форме.

**Решение:**

2.1. Пример команды резервирования и восстановления
В документации PostgreSQL для резервного копирования отдельных баз данных рекомендуется использовать утилиту pg_dump, а для восстановления — pg_restore.

1. Резервное копирование (создание дампа):

Следующая команда создает резервную копию базы данных mydb в специальном, сжатом формате custom, который лучше всего подходит для использования с pg_restore

> ```bash
>pg_dump -Fc -f mydb_backup.dump mydb
> ```
-Fc — формат custom (гибкий, сжатый).

-f mydb_backup.dump — файл, куда будет сохранен дамп.

mydb — имя базы данных для бэкапа.


2. Восстановление базы данных:

Перед восстановлением базу данных mydb (или пустую базу с таким же именем) нужно создать заново. Затем выполняется восстановление из дампа:
> ```bash
>pg_restore -d mydb mydb_backup.dump
> ```
-d mydb — имя базы данных, в которую выполняется восстановление.

**Задание 3. MySQL**
3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL.

Приведите ответ в свободной форме.

3.1. Пример команды инкрементного резервного копирования
Для инкрементного бэкапа в MySQL используется коммерческий инструмент MySQL Enterprise Backup.

Команда для создания инкрементного бэкапа выглядит так:
> ```bash
>mysqlbackup --backup-dir=/path/to/incremental_backup_dir \
>            --incremental \
>            --incremental-base=dir:/path/to/full_backup_dir \
>            backup
> ```
--backup-dir — каталог, куда будет сохранен инкрементный бэкап.

--incremental — ключевой параметр, указывающий, что бэкап инкрементный.

--incremental-base=dir:/path/to/full_backup_dir — указывает, на основе какого полного бэкапа создается этот инкрементный.

Альтернативный метод (с использованием бинарных логов):
Инкрементное восстановление также можно выполнить, используя бинарные логи (Binary Logs). Для этого их нужно регулярно архивировать с помощью mysqlbinlog с опциями --read-from-remote-server и --raw. Восстановление будет представлять собой последовательное применение этих логов после восстановления полного бэкапа.
